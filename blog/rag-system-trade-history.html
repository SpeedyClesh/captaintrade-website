How I Built a RAG System That Answers Questions About My Own Trade History

I fed three years of trade notes into a vector database and built a retrieval system on top. Now I can ask it what my edge is in London session and get a data-backed answer.

CaptainTrade
Developer · AI Engineer · Trader
February 22, 2026 Published
7 min read Read time

What is RAG and why does it matter

RAG — Retrieval-Augmented Generation — is a pattern for giving a language model access to a specific knowledge base rather than relying only on its training data. Instead of asking Claude "what's my edge in the London session?", you give it your actual trade history and let it answer from evidence.

The difference in answer quality is dramatic. A generic AI response about London session trading is useful. An AI response grounded in your personal 847-trade history, with specific win rates, average RR, and common mistakes identified from your own notes — that's a different tool entirely.

What this system does

Ask any question about your trading history in plain English. Get back a specific, evidence-based answer sourced from your actual trades, journals, and notes. No spreadsheets, no manual filtering.

Preparing the trade data

My trade data lived in three places: a Notion journal with written notes on each trade, a Google Sheet with entry/exit data and PnL, and screenshots I'd annotated over the years. I standardised everything into a simple text format — one document per trade, containing the pair, session, direction, outcome, and my written notes.

# Example trade document format
Trade ID: 2024-03-14-EURUSD-001
Pair: EUR/USD | Session: London | Direction: Short
Entry: 1.0940 | Stop: 1.0975 | TP: 1.0870
Outcome: Win | PnL: +2.1R
Notes: "Clean break and retest of 1.0950 resistance on H1.
HTF bias was bearish — weekly structure intact. Entered on
the retest with a bearish engulfing confirmation. Managed
well, closed at first target. No errors in execution."

Embedding and storing

Each trade document gets converted into a vector embedding — a mathematical representation that captures the semantic meaning of the text. Similar trades (e.g., losing London session longs during ranging markets) cluster close together in vector space, which is what enables meaningful retrieval.

I used OpenAI's text-embedding-3-small model for the embeddings and stored them in a local Chroma vector database. The entire setup — embedding 847 trades — took about 4 minutes and cost under $0.50 in API credits.

The retrieval and generation step

When I ask a question, the system converts it to a vector, finds the most semantically similar trade documents, and feeds those as context to Claude with the question. Claude then answers from the retrieved evidence rather than from general knowledge.

def query_trades(question: str, k: int = 15):
    # Convert question to embedding
    query_embedding = embed(question)

    # Retrieve most relevant trades
    results = collection.query(
        query_embeddings=[query_embedding],
        n_results=k
    )

    # Build context from retrieved trades
    context = "

".join(results['documents'][0])

    # Generate answer with Claude
    response = claude.messages.create(
        model="claude-opus-4-6",
        messages=[{
            "role": "user",
            "content": f"Based on these trades:
{context}

Answer: {question}"
        }]
    )
    return response.content[0].text

What I actually learned from it

The insights that came back were genuinely useful — and some were uncomfortable. My London session win rate is 61%. My New York session win rate is 44%. I had suspected this but never quantified it. The system confirmed it with evidence from 300+ trades.

More importantly: my losing trades share a pattern that I couldn't see when reviewing them one by one. The system identified that 73% of my losing trades involved entries taken within 20 minutes of a major news event — something I knew to avoid in theory but was clearly ignoring in practice.

How to build this yourself

  • Export your trades to a consistent text format — one document per trade with outcome and notes
  • Use the OpenAI embeddings API or a free local model like nomic-embed-text
  • Store in Chroma (local) or Pinecone (cloud) — both have free tiers
  • Build a simple Python script that takes a question, retrieves relevant trades, and calls Claude
  • Total build time: 4–6 hours. Total cost to run: near zero
